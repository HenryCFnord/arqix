# arqix 0.1.0: the tool that verifies its own announcement

Release announcements are assertions.
"Fast, reliable, well-tested" — you read it, you nod, you have no way to check any of it.
This one is different in a small but structural way: the page you are reading was staged, checked, and published by the tool it announces, from the same corpus that specifies the tool, gated by the tool's own verification loop.
If that loop were broken, this post would not have made it to the site.

arqix 0.1.0 is on [crates.io](https://crates.io/crates/arqix):

```bash
cargo install arqix
```

## The itch

Documentation drifts because nothing gates it.
Code has compilers, tests, and CI; prose has good intentions.
The requirement says one thing, the test verifies something else, the architecture document describes a system that stopped existing in March — and nobody notices, because noticing is manual labour.

arqix's bet is that the fix is not more discipline but the same machinery code already enjoys.
Markdown stays Markdown — but the frontmatter is data, relations are typed triples, and "which active test proves this requirement" becomes a query instead of a spreadsheet somebody swears they will update.

## What a day with it looks like

You write a requirement as a plain Markdown file; its frontmatter declares what it is and which story needs it.
You put one comment above the test that proves it:

```rust
// arqix:verifies REQ-04-01-03-02
```

From there, one command answers the questions that are usually folklore.
`arqix trace coverage` shows what is proven, what is merely planned, and what is uncovered.
`arqix verify` runs the whole gate — formatting, structural lint, the trace graph, and a coverage *ratchet* that lets progress grow freely but refuses to let it silently regress.
And a story in this corpus may only claim it is `done` when every requirement it carries is verified by an active test — the claim is machine-checked, not asserted in a standup.

`arqix publish site` closes the loop: it stages artefact-ready pages and hands them to whatever renders your site — no built-in renderer to outgrow, exactly as Pandoc will render the PDFs.

## The release-day story I want to tell

While writing the Quick Start for this release, I ran the five-command demo one last time — and the gate failed.

`arqix doc new requirement` scaffolds a new requirement file.
The linter rejects requirements whose lifecycle is `draft`, because this corpus decided that an unfinished obligation is not yet a requirement.
Both rules are reasonable; together they meant the tool's own scaffold could not pass the tool's own gate.
The template had been written before the lifecycle rule landed, and no human had noticed — the *demo* noticed.

That is the entire pitch, honestly.
Not that arqix (or its authors) make no mistakes — this release fixed a bug in its own template hours before shipping.
But that the mistake had nowhere to hide: the same mechanical gate that guards this corpus caught it, named the file, and blocked the claim until a failing test was written and made green.
Tools that check others should live under the checks they sell.
This one does, visibly: the [scoreboard](../reports/units/scoreboard.md) on the landing page is a generated report unit, included by the assembler and republished on every build.
Nobody edits that number, which is exactly why you can trust it when it is unflattering — and today it is unflattering: the specification is deliberately years of evenings ahead of the implementation.

## What 0.1.0 is not

It is not 1.0.0, and it does not pretend to be.
Report bundles, the policy checker, the MCP server for coding agents, PDF rendering, and the German corpus are specified, traced — and honestly listed as unverified in the scoreboard until their tests exist.
The [roadmap](../project/roadmap.md) has the order; the scoreboard has the truth; the [changelog](https://github.com/HenryCFnord/arqix/blob/main/CHANGELOG.md) has the details.

If your documentation should be part of your engineering system instead of sitting next to it: the [Quick Start](../getting-started.md) takes five minutes, and every output block in it is real.
